    How it works

    The front end runs on this static site; the CV and job description are sent to a small serverless function (a Cloudflare Worker) that holds the AI key securely and calls Google's Gemini model. Keeping the key server-side is the correct, safe pattern — it's never exposed in the browser.

    Stack: vanilla JavaScript front end · Cloudflare Worker backend · Google Gemini API · deployed free.

    Case study
    How it's built — and why

    The problem

    Most applications are read by software before a person sees them. Applicants get no feedback and no idea which keywords they missed, so they keep sending the same CV into the same filter.

    Stack & why I chose it

    Vanilla JavaScript on the front end, with a Cloudflare Worker as the back end calling Google’s Gemini model. The Worker exists for one reason: the API key. A browser-side key is visible to anyone who opens dev tools, so the key lives server-side as a Cloudflare secret and the browser only ever talks to my own /api endpoint.

    Key decisions

    The model is asked to return strict JSON for the score, matched and missing keywords, so the response can be parsed and mapped onto the UI rather than dumped as text. The cover-letter and CV-rewrite prompts carry explicit instructions never to invent employers, dates or qualifications — an AI that flatters you into a lie is worse than no tool at all. Word export is generated client-side as an HTML blob with a Word MIME type, which avoids a document-generation dependency entirely.

    Trade-offs & what's next

    It depends on one model and one vendor, and the score is indicative rather than a real ATS simulation — every ATS scores differently. Next: caching repeated CV/JD pairs to cut token cost, and a diff view showing exactly what the rewrite changed.
