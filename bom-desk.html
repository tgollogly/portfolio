BOM Desk

A data administration console for manufacturing bills of materials. It holds material and labour lines across projects, runs eight validation rules as you type, rejects bad rows at import and stamps every change to an audit log. All data below is invented for the demo.

Case study
How it's built — and why

The problem

In manufacturing, bad bill-of-materials data reaches the works order before anyone notices — a missing part code, labour booked in hours when the rate is per unit, a line still pointing at last month’s drawing revision. The cost lands downstream, in scrap and rework.

Stack & why I chose it

React via CDN with Babel in the browser, no build step, because this is a portfolio demonstrator that has to run from a static host with nothing to install. State is plain React hooks; the validation engine is pure functions over the line array, which makes each rule independently testable and easy to reason about.

Key decisions

Validation runs continuously rather than on submit, so a bad row is flagged as it is typed. Import rejects invalid rows at the boundary instead of accepting them and flagging later — letting bad data in and cleaning up afterwards is how BOMs rot. Every change writes to an append-only audit log with a timestamp, because in a controlled environment ‘who changed this and when’ is usually the first question asked.

Trade-offs & what's next

In-browser Babel is fine for a demo but would be a real build in production, and there is no persistence — refresh and the data resets. All figures are invented; this is not a deployment of any live system. Next: a proper build step, and rule severity levels so a warning and a hard rejection are distinguishable.
