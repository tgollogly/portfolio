Case study
How it's built — and why

The problem

You can read about SQL indefinitely without writing any. Most online SQL tutorials either run against a server you have to trust with your queries, or fake the results entirely.

Stack & why I chose it

SQLite compiled to WebAssembly, running in the page. That means a genuine query planner — joins, GROUP BY with HAVING, correlated subqueries, CTEs and window functions all behave exactly as they would server-side, because it is the same engine. No backend, no account, no query logs.

Key decisions

The schema is five related tables of invented haulage data rather than one flat table, because the interesting parts of SQL only appear once there are relationships to traverse. Eleven worked examples are provided as a ladder from a simple SELECT up to window functions, and every result set can be exported to CSV so the output is usable rather than just visible. Writes are permitted — the database is per-session, so there is nothing to protect.

Trade-offs & what's next

The WASM binary is a meaningful download, so the page is slower to start than a plain form. The database is gone on refresh, which is the privacy feature and the limitation. Next: query timing, an EXPLAIN QUERY PLAN view, and saving your own queries into the session.
