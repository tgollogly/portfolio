Case study
How it's built — and why

The problem

Anyone can claim their code works. Tests are how you show it — and for an early-career developer without a commercial history, demonstrating that habit matters more than another finished feature.

Stack & why I chose it

TypeScript for both the library and the test suite, compiled and executed in the page so there is nothing to install to see it run. The validators are pure functions with no side effects, which is what makes them cheap to test exhaustively.

Key decisions

The domain is deliberately unglamorous: UK postcodes, sort codes, IBAN mod-97 checksums, strict dd/mm/yyyy dates, money held as integer pence, CSV escaping. These are the things that quietly break in production — floating-point money and ambiguous date parsing cause real financial bugs. The 28 cases target edge conditions rather than happy paths. The ‘introduce a bug’ button exists because a passing suite proves nothing on its own; watching it go red and name the failing case is the part that shows it works.

Trade-offs & what's next

Coverage is deliberately narrow — this is a demonstration of testing discipline, not a library anyone should depend on. There is no property-based testing and no CI integration. Next: fuzzing the parsers, and a coverage readout alongside the results.
